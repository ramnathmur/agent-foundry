Setup — what we asked the agent to do
🤖 The agent's role

A study-prep research helper. You give it a short list of topics you want to learn about, and it builds a small reading list by finding web sources for each one.

📚 This run's topics (three of them)

python_asyncio · agent_loops · sdk_rung2_patterns — Python's async tooling, agent loop design, and one specific SDK pattern.

✅ The success rule

"Find at least 2 distinct web sources for each of the three topics, then stop." Not fuzzy — a yes-or-no test the agent could actually run on itself.

🔒 The safety cap (set by us)

Maximum 10 attempts. If the agent can't satisfy the success rule in 10 tries, it stops itself anyway. We set this limit — not the agent — so no run can spin forever.

🔧 Tools the agent could use

Two of them. A web search (returns a list of pages matching a query) and a page fetch (reads the content of one page). The model picked which to use at every step.

⚠ Important — about the "internet" it searched

This run used a small built-in fake search database, not the real web. The mock only knew certain phrases word-for-word. Searches phrased differently came back empty — even when the topic was reasonable. This matters a lot for interpreting the result.

Was this really an agent? Here's the proof.

The big question

Three things have to be true at the same time. Each one shows up in a specific line from your run.

1

Did the agent decide when it was done?

📐 [GOAL PREDICATE] is_coverage_met() → False

Yes. At every single step (all 10 of them), the agent ran its own check before deciding whether to keep going. That check is the did-it-finish trait. The fact that the answer was "no" every time doesn't change that the check fired — and was honest about the state.

2

Did the agent choose its own next move?

🎲 [MODEL DECISION] selected: search_web · alternative: fetch_page

Yes. At every step, the model — not your Python code — picked which tool to use. It had two options (search_web or fetch_page) and chose search_web all 10 times. That choice could have gone differently; the freedom was there. (Sidebar: an interesting follow-up question — given that searches kept coming back empty, should the model have tried fetch_page instead? See "What you could build next" in §10.)

3

Did the agent learn from what it saw?

🔄 [LOOP FEEDBACK] 0 new urls — model should shift topic or reformulate query next step

Partially. This is the most interesting one. After every empty search, the agent told itself "0 new urls — shift topic or reformulate" — and the model did reformulate (it tried "python asyncio tutorial event loop coroutines", then "python asyncio official documentation tutorial", then variants). So the loop fed back into the next decision. But the variations weren't different enough to match the database. A truly idle script would have repeated the exact same query 10 times; this agent kept varying. That's learning. It just wasn't enough.

✓
Verdict: this was an agent. All three core traits fired. The goal wasn't met because the world wasn't cooperating (the fake database was too strict), not because the agent failed to be agentic.
3

Why this goal counted as a real check

Verifiability

The success rule from §1 ("at least 2 sources per topic") sounds simple. The interesting question is: who decided whether the rule was satisfied? Not you watching the terminal. Not a human reviewing results afterward. The agent itself — at every single step.

A wish-based goal sounds like: "find some good sources on these topics." That's not testable. Neither the agent nor anyone reading the output could say definitively whether it was reached. An agent given a wish can't honestly stop.

This agent's goal was different. It was a computable check — a yes/no test the agent ran on itself at every step. You can see the exact moment it fired:

📐 [GOAL PREDICATE] is_coverage_met() → False

That line appeared 10 times in your run. Every step, the agent looked at its own collected sources and asked: "Do I have at least 2 distinct URLs for every topic yet?" The answer was a clean False each time — and, crucially, an honest one. The agent wasn't guessing whether it was done; it was computing whether it was done.

🎯
What this trait actually is The first core trait — "it knows what 'done' means" — isn't an attitude or a vibe. It's a function in the code that returns True or False. That function fired 10 times in your run. Even though the answer was always False, the trait itself was demonstrably present. That's the difference between an agent and a hopeful loop: an agent runs until its own verifiable test says it's done (or until the safety cap stops it). Either way, the stopping is principled.

Agent Spectrum Placement

Where this sits

A horizontal scale from "not really an agent" to "fully autonomous". Where does this one land?

⬆
WorkflowYou script every move
Simple AgentOne decision per call
Multi-Step AgentLoop + state + own choices
Autonomous AgentPlans, revises, verifies itself
📍
Landing place: between Simple Agent and Multi-Step Agent — leaning toward Multi-Step. There's a real loop (not just one call), real state (the coverage_map persisted), and real model decisions (10 different tool-and-query choices). Not yet "Multi-Step" because the plan never restructured itself — the model never said "I'm changing what I'm chasing." That comes in a future cycle.

"If this had been a workflow instead"

A workflow version would hard-code the searches: "for each topic, search Google for 'topic tutorial', take the top 2 URLs, save them, move on." Same goal, no agency. The difference would be visible right here in the output:

🎲 [MODEL DECISION] selected: search_web · alternative: fetch_page

In a workflow there would be no such line. The query would already be decided before the run started. In your run, this line appeared 10 times — each one a moment where the model exercised real choice over what to do. That is the difference, in one line.

What you could build next

Brainstorm seeds

Daily-life agents that would exercise something new beyond what Study Prep did. Each one introduces a different capability.

🌅 Morning briefing agent

You ask it "what should I know this morning?" — it pulls weather, calendar, and 2-3 headlines into a short brief. Introduces: real web tools (not a mock database), and a slightly looser goal that the agent has to interpret on its own.

🍳 Recipe-from-fridge agent

You tell it what's in your fridge — it proposes 3 meals you could make. Introduces: tools that take real input, and goal that's "good enough" rather than strictly measurable.

🧠 Conversation memory agent

An agent that remembers what you talked about yesterday and brings it up today. Introduces: memory that lives between runs — this run only kept memory within a single session (close the program and it forgot everything). The next step up is an agent whose own brain carries context forward.

📅 Calendar conflict resolver

Looks at your calendar, finds conflicts, proposes a way to resolve each one. Introduces: plan revision — when the model has to abandon its first plan because something it discovered makes it impossible.

🔍 Independent fact-checker

An agent that does some research, then a separate agent (with no shared memory) double-checks the answer. Introduces: independent verification — a check from outside the agent that did the work.
