What this agent is for

What & Why

In one paragraph

You give it a short list of topics you want to learn about — in this run it's three: python_asyncio, agent_loops, and one SDK pattern. The agent searches the web looking for reference material on each one. It stops when every topic has at least two distinct sources collected — or when it's tried 10 times without getting there.

Why this is worth building

You could write a script that runs three fixed Google searches and saves the top results. That's not an agent — it's a list of instructions. This is different: the agent decides which topic to chase next, what to search for, and when it has enough. Those three decisions live inside the running program, not in the code you wrote.

💡
The honest framing A script does what you told it. An agent does what it judges is the right next move given what it just saw. That judgment is what we're going to watch happen in real time when you run it.
2

How this agent thinks

A visual map

The agent's life is a loop of six stages. The middle four (Try → Look → Try again) repeat until the goal is met or the safety cap kicks in.

🧠
The shape worth remembering Try → Look → Try again is the loop. The model picks a move, sees what happened, picks the next move based on that. That feedback is the difference between an agent and a script that just retries the same thing.
3

What makes this an "agent" (in plain English)

The three core traits

Lots of programs run a loop. Not all of them are agents. Three things have to be true at the same time:

Trait 1

It knows what "done" means

The agent has its own did-it-finish check that it runs at every step. Not "did I do 10 steps?" — but "do I actually have what I came for?" Without this check, a loop is just an obedient repeater. With it, the loop can declare victory and walk away.

👁 Watch for this in the output: the line that starts with 📐 [GOAL PREDICATE] — fires once per step.

Trait 2

It picks its own next move

At every step, the agent looks at the situation and chooses what to do next: which topic to chase, which search to run. That choice isn't hard-coded in your Python — it's the model's call. If your code dictated every move, this would be a workflow, not an agent.

👁 Watch for this in the output: the line that starts with 🎲 [MODEL DECISION] — names the chosen tool and the alternatives.

Trait 3

It learns from what it just saw

What the agent does in step 5 depends on what came back in step 4. If the last search was empty, the agent doesn't repeat it — it shifts. That's the observe → reason → act loop: the world's reaction feeds the next decision.

👁 Watch for this in the output: the line that starts with 🔄 [LOOP FEEDBACK] — names what changed because of the previous step.

✓
All three or none If any one of these is missing, what you have is a clever script — useful, but not an agent. The Study Prep Agent has all three. We'll watch each of them fire when you run the code.

If you want to read the code

⊕ Deeper dive — skip if not curious
A short Python primer for someone who doesn't write Python daily, followed by a suggested reading order. Everything you need to recognize, not write.

Python reading primer (5 things you need to spot)

  • def name(...): — defines a function. The body is the indented block below.
  • while ...: and for ... in ...: — loops. while keeps going as long as the condition is true.
  • if .../else: — branching. Reads exactly like English.
  • print(...) — writes a line to the terminal. Everything emoji-labelled in the output comes from a print().
  • async / await — treat these as a black box. "await x" means "wait here until x finishes, then continue." That's all you need.

Suggested reading order

  1. Open main.py in PyCharm.
  2. Scroll to is_coverage_met() — the did-it-finish check. 4 lines.
  3. Then run_loop() — that's the agent loop itself. Read it top to bottom.
  4. Then call_llm() — the one function that talks to the LLM. Everything else is plain Python.
  5. Skim MOCK_SEARCH_DB at the top of the file — that's the fake "internet" this agent searches. (See §8 for why this mattered.)

What to expect when you run it

Live walkthrough

How to run it in PyCharm

  1. Open the study-prep folder as a PyCharm project. Make sure the Python interpreter is 3.12 or newer.
  2. In the terminal: pip install -r requirements.txt
  3. In the terminal: claude auth status — you should see "logged in".
  4. Right-click smoke_test.py → Run. All tests should pass green before you run the real thing.
  5. Right-click main.py → Run. Watch the terminal.

What the first few lines of output mean

🤖 [AGENT BOOT]"The agent is starting up." Confirms the SDK is ready and the topics are loaded.
🎯 [GOAL SET]"This is what 'done' means for this run." States the did-it-finish check in plain words.
📋 [PLAN GENERATED]"Here are the topics I'm going to chase." A board of three open circles, one per topic.
📍 [STEP 1]"Beginning attempt number 1 of 10."
👁 [OBSERVE]"Here's what I have so far." The agent reminds itself of the current state.
━━ [SDK →]"Handing the question over to the language model now." This is the line where your code stops thinking and the LLM starts.
🎲 [MODEL DECISION]"The model picked this tool over the alternatives." This is the model's own choice — the second core trait.
🔧 [TOOL CALL]"Calling the chosen tool with these arguments."
📥 [TOOL RESULT]"Here's what came back."
🔄 [LOOP FEEDBACK]"Based on what I just saw, here's what I'll change next step." This is the third core trait — learning from what it saw.
📐 [GOAL PREDICATE]"Am I done? Running my own check now." The did-it-finish check fires here — the first core trait.
🏁 [GOAL MET] or [EXIT: cap reached]"Stopping. Here's why." The safety cap means the agent will never run forever.
📍
Read the labels, not the code Every emoji-labelled line is a teaching marker. You don't need to read the surrounding Python — the labels tell you exactly which part of the agent's life you're watching.

Things worth noticing during the run

Insights

The agent doesn't remember conversations

Every time you see [SDK →], the LLM starts fresh. It doesn't remember anything from the previous step's chat. The way the agent "remembers" anything is that your Python re-shows it the current state at every step. The model itself is forgetful by design at this stage.

A future build (next cycle) introduces sessions — that's when the LLM itself will start carrying context across calls.

The log file keeps every run

After the run, you'll find study-prep_run_output.log in the same folder. It captures everything the terminal showed. Every future run appends to the same file — so you can compare how the agent behaved differently across runs once we build more agents.

The model talks a lot — but only the JSON matters

You'll see the model emit text blocks, then a JSON line like {"tool": "search_web", "query": "..."}. Only the JSON is parsed and acted on. The prose is the model's thinking on the way to that decision.

8

Watch out for these

Gotchas
Gotcha 1

The mock "internet" only knows certain queries word-for-word

This agent searches a fake search database we built in. It uses exact matching — so if the model searches "python asyncio tutorial event loop coroutines" but the database only has "python asyncio tutorial", every result comes back empty. The agent will keep trying, but it won't find anything. This is what happened on the first run — see Part 2 (after you've run it) for the full story.

Gotcha 2

Empty results aren't errors — they're information

When a search returns nothing, the agent doesn't crash. It treats "zero results" as a signal to try something different. Whether it actually does try something different is what the third core trait (learning from what it saw) is about. Watch the [LOOP FEEDBACK] line carefully — it tells you whether the agent changed strategy.

Gotcha 3

The safety cap WILL fire if the goal can't be reached

At 10 iterations, the agent stops regardless. This isn't a bug — it's the principled stop trait working correctly. If you ever see the cap fire, ask: was the goal unreachable, or was the agent making bad choices? Both have different fixes.

Gotcha 4

"Plan revised" doesn't show up in this agent

You won't see a [PLAN REVISED] line in this run. That's not a bug — the plan here is simple and linear (the three topics never change). You'll first see plan revision in a future cycle, when the model encounters a surprise and restructures what it's chasing.

Gotcha 5

The same URL might show up twice — the agent has a dedup guard

If the model asks the same search twice and the same URL comes back, the agent checks before saving. You'll see [DUPLICATE URL DETECTED] if that happens.

Who's making the decisions

Control plane

Every step in the agent's life is a decision. Some are made by your code (deterministic, predictable). Some are made by the model (judged in the moment). Knowing which is which is the single most important thing you'll learn from reading agent code.

DecisionWho decidesWhy this matters
How many tries before giving upYour code (MAX_ITERATIONS = 10)You set the safety boundary. Even a confused agent can't burn unlimited credit.
What "done" meansYour code (is_coverage_met())You define success. The model can't move the goalposts mid-run.
Which topic to chase this stepThe modelThis is the freedom you're handing over. The model judges based on what it sees.
Which tool to use (search vs fetch)The modelSame: model picks, your code dispatches.
What search query to sendThe modelSame: free-form text the model invents.
Whether to stop nowThe did-it-finish check (your code) AND the safety cap (your code)The model proposes, the code disposes. The agent itself can't keep going past either limit.
🔍
If you removed the model's freedom Hard-code the topic order, hard-code the search queries, and what you have is a workflow — useful, predictable, not agentic. The freedom you're giving the model is exactly what makes this an agent.

A few self-check questions

Before you run

Click an answer to reveal whether it's right and why.

1. The agent runs 10 attempts and stops with the goal unmet. What does that prove?
✓ Correct: B. Hitting the cap is the principled-stop trait working. The agent is supposed to stop itself rather than burn through unlimited tries. Whether the goal was actually reachable is a separate question.
2. Which line in the output is the model's own choice?
✓ Correct: B. The model-decision line names the tool the model picked and what alternatives it had. [GOAL SET] is your code declaring the goal up front, and [TERMINATION CHECK] is your code checking the limits.
3. If you removed the while loop and just called the model once, what would you have?
✓ Correct: C. The loop is where "learning from what it saw" happens. Remove it and you have a one-shot search call dressed up. The loop + did-it-finish check + model decision is what makes the difference.
4. The coverage_map dictionary is re-shown to the model at the top of every step. Why?
✓ Correct: B. At this stage of learning, each call to the model is stateless — fresh slate. The way the agent maintains continuity is by re-injecting state. A future cycle introduces sessions, where the LLM itself remembers.
