What It Achieves

Practical understanding through construction

This project does not teach agents through slides or reading. It teaches them by having you build one in every session. By cycle 5, you will have built five increasingly capable agents and will be able to explain, from code, exactly why each one is an agent and not just a script.

🎯

Agency that is provable, not asserted

Every agent is accepted or rejected by five formal Agency Gates (G1–G5). You learn to apply the same gates, not just hear about them.

🪜

A cumulative SDK concept ladder

Each new agent introduces at least one SDK concept the previous one did not. You climb from basic query() calls to subagents and context compaction, one rung per cycle.

👁

Visible runtime behavior

Every agent prints 24+ labeled output lines — including the Python↔LLM boundary, model decisions, goal predicate evaluations, and loop feedback. You watch the agent think.

📚

Two learning artifacts per agent

A pre-run HTML guide (why it's an agent) and a post-run HTML insights file (proof from execution). Together they complete the learning loop.

🔁

Cross-cycle retention

A registry accumulates every agent built, concept introduced, and gap identified. Each new brainstorm session builds on the last. Nothing is lost between sessions.

📐

A framework you can apply

The 9-trait framework and Agency Gates are not invented for this project — they are a portable vocabulary you can use to evaluate any agent you encounter in the wild.

How It Is Structured

The 8-rung SDK concept ladder

Each cycle introduces at least one new rung. Rungs are tracked in the project registry so the brainstorm always builds forward. Click any rung to see what it introduces and when you will encounter it.

  • 1 query() + message anatomy Cycle 1 ▶

    The basic call: how to send a prompt, receive messages, and read the usage object. You see SystemMessage, AssistantMessage, and ResultMessage for the first time. The agent loop uses query() on every iteration.

  • 2 Goal predicate + agent loop + circuit breaker Cycle 1 ▶

    The foundation of agency: goal_met(state) → bool evaluated every iteration (G1), the while loop that runs until it returns True (G3), and a hard iteration cap that prevents infinite loops (G4).

  • 3 ClaudeSDKClient + session_id capture and resume Cycle 2 ▶

    Multi-turn sessions where the model's context persists across SDK calls. Cycle 1 uses stateless query() — each call starts fresh. Rung 3 makes context cumulative. You will understand why this matters before you see the code.

  • 4 Custom tools — @tool, model-owned selection (G2) Cycle 3 ▶

    The model decides which tool to call — not a Python if chain. This is where Agency Gate G2 (model-owned decision) becomes fully demonstrable. The [MODEL DECISION] output line acquires a new meaning at this rung.

  • 5 allowed_tools / permission_mode Cycle 3–4 ▶

    Controlling what the model is permitted to do. Autonomy is not a binary — you learn to dial it precisely. The four documented permission modes each produce a different behavior.

  • 6 Hooks — PreToolUse guard, PostToolUse audit Cycle 4 ▶

    Hooks intercept tool calls before and after execution. A PreToolUse guard can block unsafe operations; a PostToolUse audit writes a JSONL span trace. You can see every tool invocation logged, not just printed.

  • 7 Independent verifier — separate critic call, fresh context (G5) Cycle 4–5 ▶

    Agency Gate G5: verification is independent. A second SDK call with a fresh context checks the first agent's work. This is where trust and reliability become architectural concerns, not afterthoughts.

  • 8 Subagents + context compaction Cycle 5 ▶

    Each subagent has its own context window. Context compaction prevents runaway token accumulation. This is the frontier: orchestrating multiple agents, each with bounded context, working toward a shared goal.

The Student Experience

What one cycle actually looks like

A cycle takes one Claude Desktop session. The duration depends on how deeply you engage with the Professor sessions — the generation itself typically takes 15–20 minutes. Below is what you do and see at each phase.

Phase A

You describe a daily-life idea

Claude asks a short sequence of structured questions — domain, complexity preference, what you want to learn. It proposes 3–4 candidates, each with a Learning Position label (FORWARD, FOUNDATIONAL, etc.) and a G1–G5 gate verdict. You choose one and lock it with an explicit command. Nothing is written before you lock.

Phase B

Claude shows a mini-spec and waits for confirmation

Before any code is written, you see the goal predicate, tools, loop structure, and termination conditions in plain English. You confirm or adjust. This is the last human checkpoint before generation begins.

Phase C

Claude generates all 8 files and runs its own QA

Code is generated, the smoke test is run in Claude's sandbox (LLM mocked), failures are repaired and narrated. You watch a feedback loop work before you read one. When Claude says "green," the smoke test has actually passed — not been skipped.

Phase C — Read first

You read the learning guide (Part 1) before opening the code

The Part 1 HTML is delivered before main.py. It explains why this agent is an agent, maps each trait to a function, shows you what the runtime output will look like, and includes a Python Reading Primer so you can follow the code even without a Python background.

Phase F — Skippable

The Professor session (pre-run)

An interactive ~10-minute conversation. The Professor asks 2–3 Socratic questions anchored to the code you just read. On a wrong answer, you get one counter-question, then a targeted explanation pointing to the exact line. No scoring. No shame. The session ends with a note on what to watch in your first run.

Your run

You run main.py in PyCharm

The agent prints its full operation to the console — every SDK call, every model decision, every goal predicate evaluation. The output is saved to a .log file automatically. The brief (or whatever the agent was built to produce) appears at the end.

Phase F2 — Skippable

The Professor session (post-run)

You say "I ran it." The Professor asks 3–5 probes anchored to specific lines from your actual runtime output — not the code. This is typically more useful than Phase F because you have evidence to point to.

Phase C — Post-run

Claude generates the learning insights file (Part 2)

Using your runtime log as evidence, Claude produces the Part 2 HTML. It proves — from quoted output lines — that G1, G2, and G3 were satisfied. Flashcards and MCQs reinforce what you just saw. Brainstorm seeds show you what to build next.

Salient Features

What makes this different from a course or tutorial

The key difference: you are not consuming content about agents. You are building them, watching them run, and being asked to explain why they behave the way they do. The code is the curriculum; the runtime output is the exam.

🚦 Agency Gates — not just a checklist

Five formal pass/fail gates (G1–G5) determine whether what Claude generated is actually an agent. If G1, G2, or G3 do not pass, the artifact is called a workflow and a fix is proposed. You learn to apply the same gates yourself by watching them applied.

📺 24-element runtime output standard

Every generated agent prints the same set of labeled elements — including the Python↔LLM boundary brackets [SDK →]/[← SDK], the model's tool selection decision, and the goal predicate result every iteration. The output is a teaching artifact, not a debug log.

🎓 Two-phase Professor persona

A pre-run session (Phase F) sets up what to watch. A post-run session (Phase F2) asks you to explain what you actually saw. Both are anchored to specific artifacts — code lines or output lines — not abstract concepts. Both are skippable.

🔎 Honest QA narrated in real time

When the smoke test fails, Claude narrates what failed, what changed, and why. You see a feedback loop work before you read one. The repair loop is capped at 3 attempts — after which Claude tells you what it could not fix.

🧭 Learning Position labels

Every brainstorm candidate is labeled FORWARD (new SDK rung), FOUNDATIONAL (targets your weakest gap), LATERAL-RIGHT (new domain), or DIAGNOSTIC (deliberately built to fail a gate). You always know whether a cycle is advancing, consolidating, or repairing your understanding.

📋 Persistent cross-session memory

A JSON registry remembers every agent built, every concept introduced, every gap identified in Professor sessions. The next brainstorm reads it. No repetition; no forgetting; no starting from scratch.

Tips and Best Practices

What makes the difference between a productive cycle and a frustrating one

Read Part 1 before running. Every time. +

The runtime output is 24+ labeled lines per iteration. If you open PyCharm first, you will see 80+ lines of structured output before the agent's actual result appears. Without Part 1's annotated output guide, this is overwhelming. With it, each line is recognizable.

Part 1 has a "Python Reading Primer" subsection specifically for readers who do not write Python. Five concepts in five minutes. It covers everything you need to follow the guided reading order.

Do not skip Phase F2. It is usually better than Phase F. +

The pre-run Professor session (Phase F) is useful. The post-run session (Phase F2) is where understanding consolidates. Every probe in F2 starts with "find this line in your log" — you are explaining real evidence, not recalling a definition. The difference is measurable.

You can skip Phase F entirely and go straight to running. Phase F2 is the one to protect.

Watch the [SDK →] / [← SDK] brackets on your first run. +

Everything between those two lines is LLM time. The elapsed time shown on the closing bracket (← SDK] 1.2s) is the model thinking. The token count is what was consumed from your monthly credit allocation.

On first run, find the bracket pair and ask: what Python code runs between those lines? The answer is: none. That gap is where the agent is not under Python's control.

The [MODEL DECISION] line is the most important one. +

Find [MODEL DECISION] selected: X in your output. Ask: could Python have predicted what X would be, given only the code? If the answer is no — because it depends on the content the model saw at runtime — that is Agency Gate G2 satisfied.

In a workflow, every branch is predetermined by the code. In an agent, at least one branch is owned by the model at runtime. The [MODEL DECISION] line makes this visible.

Run the agent more than once and compare the logs. +

The log file appends — it never overwrites. After two or three runs, you can compare the [MODEL DECISION] lines across runs. If the model made different decisions with different inputs, you are watching G2 in action across multiple executions.

By cycle 3, your log file is a record of how the agent behaves across different contexts. This is the compounding value of the tee mechanism.

Use INSIGHTS.md as your learning pulse check. +

INSIGHTS.md is regenerated at the end of every cycle. It shows your active gaps (concepts that came up weak), which SDK rungs you have introduced vs. not yet encountered, and which domains you have built in. At any point you can open it and see where you are in the curriculum.

The "Recommended focus for next cycle" section (§6) is the most actionable part — it names the specific gap to repair and which rung to introduce next.

Honest Caveats

What this project is not, and where it may fall short

These are not disclaimers added to hedge a marketing claim. They are accurate descriptions of real constraints. Read them before starting.

⚠️

This is not a Python course

You will read Python code, not write it. The Part 1 guide includes a 5-concept reading primer. If you want to modify the generated agents, you will need Python fundamentals from another source.

⚠️

The runtime trace is teaching, not production output

24 labeled lines per iteration is noisy for production use. The output standard exists because this is a learning system. Agents built for real use would have quieter, more targeted logging.

⚠️

Claude Max plan required

The agents authenticate via the Claude Code CLI and your Max plan subscription. There is no API key option. If you do not have a Max plan, the agents will not run.

⚠️

Requires Windows + PyCharm

Generated agents are tested on Windows with Python 3.12+. They are standard asyncio Python and would likely work on macOS, but this is not tested or supported in the spec.

⚠️

The Professor sessions require engagement

Both Phase F and Phase F2 are Socratic — you are asked questions, not given answers. If you skip both sessions every cycle, the learning artifacts (HTML files, registry) still accumulate, but the retention benefits of active recall do not.

⚠️

This is one student's learning system

It is designed for a single user (Ram), a single machine, and a specific learning style. It is not a product for others, not a course platform, and not maintained as open-source software. If you adapt it for your own use, you are on your own.
